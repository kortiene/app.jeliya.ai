# Internationalization

The language plan of record, and the engineering rules that keep the web
client translatable. Terminology and French scoping live in
[`glossary-fr.md`](glossary-fr.md); this file records the decisions and the
mechanics.

Jeliya ships one client, the React web client in `ui/`. Everything below is
about that client. The daemon is not a localization surface (glossary
decision 1).

## Decisions

- **A language ships complete or not at all.** The workbench shows the room
  rail, the timeline, and the inspector at once, so a partially translated
  catalog would put two languages in one window. `SUPPORTED_LOCALES` in
  `ui/src/l10n/locale.ts` is the list of languages with a complete catalog;
  a tag outside it falls back rather than rendering half-translated.
- **Bambara (`bm`) is a fast-follow** via community translation. The hosting
  mechanism is still open: the catalog is a typed TypeScript module, not a
  translation-tool file format, so a translator-facing seam has to be chosen
  and built before community work can start. CLDR gives `bm` a single plural
  category (no plural forms needed) but below-basic coverage — under a `bm`
  text locale the formatting locale stays `fr`/`en`, which glossary decision 4
  makes expressible.
- **N'Ko (`nqo`) is pending a rendering test** — RTL script with cursive
  joining, so it needs a bundled webfont and a full RTL layout audit before it
  can be considered. **`bm_Nkoo` is not in ICU — never target it as a
  locale.**
- **Daemon/CLI output stays English** (glossary decision 1). The client maps
  `{code, message, hint}` to translated copy; raw daemon text is only ever the
  collapsed "Technical details" disclosure, the Settings diagnostics card, and
  the diagnostics report (support artifacts).
- **Text locale ≠ formatting locale** from day one (glossary decision 4).

## Engineering rules

[`scripts/check-ui-i18n.mjs`](../scripts/check-ui-i18n.mjs) enforces rule 1
mechanically: catalog key parity, no empty value, no French value still
identical to its English one, French typography, and no user-visible literal
in the scanned components. The other six rules are review obligations — the
literal scan catches some violations of rules 2 and 3 as a side effect, but
nothing checks them directly. Run the gate from the repository root:

```sh
node scripts/check-ui-i18n.mjs
```

1. **All user-visible copy lives in the catalog.**
   `ui/src/l10n/catalog.ts` declares the shape — stable `<area><Key>` names in
   lowerCamelCase — and `en.ts` and `fr.ts` implement it. Because both are
   typed `Catalog`, a missing key is a compile error; the gate covers what
   types cannot see, rejecting an empty value, a French value still identical
   to its English one, and a user-visible literal in `ui/src/App.tsx` or
   `ui/src/components/` that never reached the catalog. Non-copy strings
   (glyphs, URLs, shell commands, wire-format examples, the brand, language
   endonyms) live in `ui/src/l10n/tokens.ts` and never reach translators.
   Deliberate exceptions carry an `// i18n-exempt: <reason>` marker, or an
   entry in the gate's `EXEMPT_FILES` map with the decision behind it.
   Components resolve copy AT RENDER TIME via `useStrings()` — never cached
   into state or a module constant, because that is what makes a language
   switch apply live.
2. **No sentence assembly in component trees.** A sentence with styled or
   interactive segments is ONE catalog message whose `{slot}` placeholders the
   call site fills, rendered via `ui/src/l10n/template.tsx` — translators
   reorder slots like any placeholder. Lists of independent facts may be
   joined with the decorative `Punct.metaSep`; sentences may not.
3. **No wire values as display text.** Protocol enums (roles, member statuses,
   peer paths, daemon modes, connection states) render through
   `ui/src/l10n/wireDisplay.ts`; error copy through `friendlyError` in
   `ui/src/l10n/errorDisplay.ts` — resolved per render. Unknown values pass
   through raw (forward compat). Display labels and wire values are never the
   same constant.
4. **All display formatting goes through `useFormats()`**
   (`ui/src/l10n/formats.ts`): bytes, clock, day labels, percent, counts, and
   relative time. Numeric and calendar conventions use the EXPLICIT formatting
   locale; vocabulary such as byte units, Today, and "ago" follows the text
   catalog. Never call `Intl` or `toLocaleString` at a call site, and never
   format a number or date inline.
5. **The transport and protocol layer composes no user-visible English.**
   `ui/src/lib/` emits structured facts — `join.ts` reports
   `phase`/`attempt`/`retryDelayMs` rather than a narrated sentence — and the
   components narrate them through the catalog. `ui/src/lib/format.ts` is an
   English-token contract in the other direction: `labelTone`/`prettyLabel`
   map wire values to tone classes, and those tokens are protocol, not copy.
   Exception and error text is technical detail, not UI copy.
6. **Tests assert copy via the shared `en` catalog**, never a literal, so
   translation work never breaks them:

   ```ts
   import { en } from '../src/l10n/en';
   expect(screen.getByRole('button', { name: en.roomsCreate })).toBeVisible();
   ```

   Fixture *data* — room names, message bodies, file names — stays literal. It
   is data, not copy, and pinning it is the point.
7. **All-caps treatments use `toUpperCase()` at render time** on an
   already-translated string. That is Unicode-correct and safe for the target
   locales (French and Bambara keep their accents); if a locale with
   locale-dependent casing (Turkish, Greek) is ever added, casing must move
   into the catalog.

## Locale resolution and switching

Settings persists two independent browser preferences:

- `jeliya.textLocale`: unset means follow the first supported primary language
  in `navigator.languages`, falling back to English. `fr-CA` and `fr-BE` both
  match the French catalog on the primary subtag.
- `jeliya.formattingLocale`: unset means follow the first platform locale that
  `Intl` supports, even when that locale has no text catalog; if none is
  usable, it falls back to the resolved text locale.

Both preferences apply live. A browser `languagechange` event re-resolves an
unset preference without a reload. The Settings language card writes both, and
names each language by its endonym from `LANGUAGE_NAMES` in `tokens.ts` — a
picker showing a bare ISO code is a bug.

## Status and what is next

English and French are both complete catalogs and both selectable. The French
translation follows [`glossary-fr.md`](glossary-fr.md): Tier 1 vocabulary,
Tier 2 wire tokens verbatim, the Tier 3 brand line as the onboarding tagline,
and decision 7 typography (U+202F before `; ! ? %` and inside « », U+00A0
before `:`, U+2019, U+2026, sentence case, vouvoiement, octets o/Ko/Mo/Go).
`scripts/check-ui-i18n.mjs` blocks missing, empty, untranslated, or
typographically invalid entries and any new user-visible component literal
outside the catalog; `npm run i18n:report` from `ui/` is a non-failing
inventory view of that same literal scan. Layout under both languages is
exercised by `ui/e2e/i18n-layout.spec.ts` at the 320px reflow target.

Not done, in order:

1. **A translator-facing seam.** Community translation needs a file format a
   translation platform can read and a round-trip back into `fr.ts`-shaped
   modules. Nothing in the tree does this yet.
2. **Bambara.** Blocked on the seam above. Standard orthography needs ɛ ɔ ɲ ŋ;
   smoke-test the mono stack before shipping it. Adding a locale also means
   adding its endonym ('Bamanankan') to `LANGUAGE_NAMES`.
3. **N'Ko groundwork.** Rendering test, bundled webfont, RTL audit — none
   started.
4. **Discoverability for francophone users.** There is still no `README.fr.md`
   or "Voir aussi" pointer from the main README, so a French speaker has no
   signposted way in.
